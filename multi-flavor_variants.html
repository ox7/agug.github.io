Multi-flavor variants

多个flavor变体

当想要创建基于多个标准的app不同的系列版本。
举个栗子：支持Google Play的multi-apk支持4个不同的过滤器。为每个过滤器都创建不同的APK，并用Product Flavors满足不同维度的需求。

考虑到一个游戏app有demo和paid版本，且multi-apk想支持ABI过滤器。既3个ABIs和2个应用版本，将会生成6个APK。

而且3个ABI的paid版本源码是一致的，而不是简单的创建6个flavor。

此外，2个分辨率的flavor和variant可以自动生成所有可能的组合变体。

使用Flavor Dimensions来实现这些特性：

android {
    ...


    flavorDimensions "abi", "version"


    productFlavors {
        freeapp {
            dimension "version"
            ...
        }

        paidapp {
            dimension "version"
            ...
        }


        arm {
            dimension "abi"
            ...
        }

        mips {
            dimension "abi"
            ...
        }

        x86 {
            dimension "abi"
            ...
        }
    }
}

android.flavorDimensions数组定义了可用的维度和顺序，每个被定义的Product Flavor都分配了一个维度。

所有可用的组合变体如下：

  • x86-freeapp-debug
  • x86-freeapp-release
  • arm-freeapp-debug
  • arm-freeapp-release
  • mips-freeapp-debug
  • mips-freeapp-release
  • x86-paidapp-debug
  • x86-paidapp-release
  • arm-paidapp-debug
  • arm-paidapp-release
  • mips-paidapp-debug
  • mips-paidapp-release

定义在android.flavorDimensions的维度规则是非常重要的。

每个变体都被多个Product Flavor对象来配置：

  • android.defaultConfig
  • One from the abi dimension
  • One from the version dimension

这些维度的优先级规则如下：

abi > version > defaultConfig

多个flavor的工程还会有额外的Sourcesets，和variant的Sourcesets一样但是没有Build Type：

  • android.sourceSets.x86Freeapp
    Location src/x86Freeapp/
  • android.sourceSets.armPaidapp
    Location src/armPaidapp/
  • etc...

这些都是可以在flavor组合等级中自定义的，它们的优先级高于基本的flavor Sourcesets，但是低于build type的Sourcesets优先级。
