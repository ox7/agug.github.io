Gradle Android Plugin Learn

Build Types

构建类型

默认的，Android plugin会自动为工程建立两个版本的应用一个debug和一个release版本。它们之间不同之处大多在，其一debug应用，其二如何构建一个签名APK。debug版本的APK签名是系统自动创建的，relase版本在构建期间不会签名而是在构建之后对APK签名。

这些配置是通过BuildType类来执行的，默认的会创建两个这个类的实例，一个是debug一个是release，Android plugin允许创建和这两个实例一样的自定义Build Types, 在BuildType的DSL容器中操作：

android {
    buildTypes {
        debug {
            applicationIdSuffix ".debug"
        }


        jnidebug {
            initWith(buildTypes.debug)
            packageNameSuffix ".jnidebug"
            jniDebuggable true
        }
    }
}

上面的构建片段将会得到：

  • 配置默认的debug构建类型
    • 更改debug类型的APK包名加.debug后缀以便于debug和release类型的APK可以在同一设备上安装
  • 创建一个叫jnidebug的构建类型，并配置成debug类型的副本
  • 打开jnidebug类型构建的JNI模块的debug功能，并将添加一个不同的包名后缀

创建一个新的构建类型和在BuildType容器中创建一个新元素一样容易，无论是调用initWith()方法还是用闭包都可以配置轻易配置。

查看DSL Reference可以得到构建类型所有可配置的属性。

通过修改额外的配置属性，Build Types可以被用于指导不同源码和资源。每个构建类型，都会创建一个新匹配的sourceSet，默认路径为src/<buildtypename>/,比如src/debug/java目录可以添加为Debug APK而编译的源码文件。也就是说Build Type的名字不能是main或者androidTest，必须是唯一的。

和任一构建类型一样，sourceSet可以被重定义：

    android {
        sourceSets.jnidebug.setRoot('foo/jnidebug')
    }

另外，针对每个Build Types，都会创建一个新的assemble<BuildTypeName>任务，例如：assembleDebug。这也是上面已经提及的assembleDebug和assembleRelease任务的由来。当debug和release Build Types被预创建时，它们的任务也会自动被创建。根据这个规则，上述的build.gradle片段不仅将会创建assembleJnidebug任务，并且assemble任务也会像依赖assembleDebug和assembleRelease任务一样依赖它。

可能使用到得场景：

  • 仅在debug模式才用到的权限，release模式没有
  • 为debugging自定义的一些实现
  • 为debug模式采用的不同资源

BuildType的源码和资源会被如下方式使用：

  • manifest会被合并到主App的manifest中
  • 代码被认为另外的源码文件夹
  • 资源会覆盖主资源，并替换已有值
